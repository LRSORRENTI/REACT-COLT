What is React?

React is a popular front-end framework / library that's developed and sponsored by Facebook

The main point of React is to:

1 Make it easy to create reusable "view components"
2 These "encapsulate logic and HTML into a class"
3 Often makes it easier to build modular applications

React apps are made up of smaller components and built up until we have a full size application, we use the tools React provides to build these components

Front end frameworks:

Front end frameworks are larger than a library, for example we can have a library for setting date and time, and we plug that library into the framework, the framework provides the structure / blueprint, and you add the rest

They are often 'opinionated' meaning you have to follow the rules of the framework, which can often be restricting depending on what your goals are. This is often referred to as 'inversion of control', with a library, you're in control of the structure, with a framework, the framework has control

Frameworks often provide code for reuse inside the framework, as well as providing templating of HTML

The main frameworks are:

  • Angular
  • Vue
  • Ember
  • React??

There are dozens of frameworks out there, those 4 are the most popular

React does less 'out of the box' than a tradtional framework, but when most developers utilize React, they're also using React specific tools alongside

Things like: React Router, Redux, Remix, there's a lot of of React tools available for use, and there's almost a 'standardized stack' for developing React

Library or Framework?

So at face value React is a library, on it's own.

However, few people use React just by itself, it's almost always combined with things like React Router, Webpack etc..

So when React is used in conjunction with tools like React Router, Redux, Webpack, React becomes an 'Ecosystem Framework'

But there's also Next.js which is an actual framework built around React

Components

Let's talk about components, components are the main building block of React, React is about creating these components and structuring how these components communicate with each other, pass data bewteen one another

Components let you split the User Interface into independent, reuasble peices, and think about each peice in isolation

Components usually combine peices of UI and JavaScript logic, for example the first component we'll look at is the rating component, where you can click a number of stars out of four or five and the page will update based on the selection

Components usually have classes that will render themselves into HTML on the page

Looking at a large app

If we're working on a large project, our file structure will have many different JavaScript files, 10-30 or however many, but each of our components is localized inside it's own unique JavaScript file, so our SRC folder will have a LOT of files inside

For example, we can have a navbar component, that we use in different places in our react app, but the navbar can be a little different depending on where it is, so we have our navbar js file component, and we can reuse it, tweak it, and change it depending on where and how we want to

The main takeaway is that in a react app, we can have components that structure our webpage, we can have a default slate, like a navbar component, and we can take that navbar component and customize it depending on where the user is in our application, so we could have one central navbar uncustomized, then take that navbar component and use it in 3 or however many unique locations, changing how the navbar looks and functions each time

Function component

We just wrote our first class component, but we can also write function components

A Class component is the traditional React component, we write our logic inside of a JS class, and we MUST include the render() method

Function components are usually used for simpler, more basic components

They don't have the same capabilities as class components, and there's no render method needed, just the function return statement

function Dog()

{

return '< p > German Shepherd, Cava-Corgi < / p >''

}

So what's the main difference?

Both can accept props and render content to the page

Historically function components couldn't utitlize 'state' and 'lifecycle methods'

With the introduction of Hooks, we can write full-featured function components, which will dive into later!

For now though, we'll focus on the traditional class based components, that way we can only worry about one kind of syntax for now, and of course, we'll deep dive into Hooks later in the course!

In summary we can render things to our page with both class, and function components, but function components are usually utilized for simpler tasks

Introducing: JSX

This section we'll learn about JSX which stands for JavaScript Syntax Xtension, or JavaScript + XML

JSX is the tool that allows us to write HTML looking code inside of a JavaScript file, there's a lot going on behind the scenes with BABEL turning the HTML characters in JSX to valid JS code

It's possible to create React applications without JSX but it's more time-consuming and results in more code needing to be written

JSX is NOT legal JavaScript

it has to be transpiled to valid JavaScript using Babel

JSX Rules

JSX is more strict than HTML -- elements must either have an explicit closing tag, or be self-closed, the '/' is critical, otherwise we'll always get a syntax error

And remember is we want to render multiple elements in a single component, they need to be inside of a div tag, or a section tags

Embedding JS Expressions in JSX

We can embed JavaScript functions or other data-types using curly braces nested inside our html tag and inside render(), this works just like JavaScript but passing it in the render as an expression

Conditionals in JSX

We can also add conditional logic inside of our JSX, how we can display things selectively depending on certain conditions

The exercise showed how we can use if/else, ternary operators to display messaging depending on the condition, for smaller conditional checks the ternary works well, for larger conditional checks implementing if /else works

There are other ways to structure these conditional checks to inject data into our webpage we'll see later

How to Layout a React Application

In this section we'll learn how to properly structure the files inside of a React application

So far we HAVEN'T been following it, we've been creating a single component inside of an index.js file

It's proper to create a component in it's own file, with a filename that is obvious at a glance what the component is.

So if we have a greeting component, we create a file called greet.js, if we have a number picker, we create a numpicker.js file

One component per file. If our app has 5 compoments, we have 5 js files, if it has 10 components we have 10 js files, all should have obvious naming as to what the component is doing,

App

Then after we have our components in separate files that have obvious names, we create our App component, and our App component is where we utilize our separate js component files into a single element

So we'd have our main App component that will be the top-level component, which renders all of the applications individual / separate compoments

This way it's easier to read the App code, and it's almost always the only thing rendered in the main index.js

And like always the ORDER OF THE SCRIPT TAGS are vitally important

So when we make a react application, we'll always separate our components into their own unique js files, then in our main index.js, where we have our 'class App extends React.Component', inside the render we add our components that we want to render, keeping them separate

Intro to REACT PROPS AKA PROPERTIES

Props or properties allow us to make components more customizable, configurable AND resuable!

For example, with our last Greet component, we can add props that will be given for 'to': or who we are greeting, and 'from': who is greeting us. Hello Luke, From Rene

Just finished the exercise, we can use '{this.prop.key}' to like in JS template literals, pass in values, we can also to shorten it do:

const prop = this.props, then '{this.prop.key}' becomes:

{props.key}

And the '.key' being what we defined inside the main index.js, render function as key value pairs, like inside of an object

< Hello to:"Luke" from:"Rene", so like an object, 'to' is the key, and "Luke" is the value

Props are immutable?

This is crucial! Properties are for configuring your component, but these properties are immutable, they cannot change

That doesn't mean components can't be altered, we just can't do it through props

This will be more obvious when we get into things like 'state', rendering and re-rendering

Other types of Props

So far all our properties have been strings, but we pass in other data-types as well, numbers, booleans, arrays using, curly braces to embed them

We can also pass in images, using an img tag as a prop, then in the component file we do : < img src={this.props.img} / >
